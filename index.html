The fulfilment flow, on Temporal

A parent RequestWorkflow spawns a child LinePlanWorkflow per line; task execution and rollback run as activities that call external app capabilities. In-flight state lives in workflow history — the state machines run inside the workflows. These are the real call sequences, traced from the executor/ code.

Entry 1 · Create

CreateFulfilment → start workflow → execute

The API validates, persists the request (idempotent on tenant + key), plans the lines, marks the request RUNNING, then starts the RequestWorkflow and returns immediately — execution runs durably in the background. The parent spawns a child per line; each child dispatches recipe steps as ExecuteTask activities. Mode is completion timing: SYNC returns inline, ASYNC kicks off external work and parks. Invocation is orthogonal — a step that declares an app is dispatched to it; an app-less step invokes nothing and parks for an operator. Parked tasks await a task_callback signal.

Entry 2 · Report

An external callback resumes a parked task

A parked task waits on the task_callback signal — an app callback (ASYNC) or an operator completing an app-less step. The caller is outside the engine. ReportTaskResult checks tenant first (a foreign task reads as not-found — keys can't be probed), runs the idempotency ladder, settles the row synchronously (status+version CAS), then signals the child LinePlanWorkflow directly, which applies the result and dispatches any unlocked successors.

Admin · Cancel / Compensate

Teardown as a durable workflow

Both admin paths are eventually consistent: the API CASes the request into a transitional status — CANCELLING (compensate=false, completed work kept) or COMPENSATING (default, roll back) — and starts teardown-«id»; the workflow owns everything after, so a crash resumes instead of stranding a half-cancelled request. Rollback reuses the same CompensationWorkflow as the organic failure saga, walking completed tasks in reverse order. A force-delete of an in-flight request triggers this same cancel first — no orphaned workflows.

Ops · Recovery sweep

The sweep closes the crash window

One gap the durable teardown can’t close itself: a crash between the accept-CAS and the workflow start leaves intent in Mongo that Temporal never heard about; a teardown that exhausts its retries leaves a Failed workflow. Cloud Scheduler invokes POST /internal/fulfilment/teardown-sweep every ~5 min: it lists requests stuck in CANCELLING/COMPENSATING past TEARDOWN_SWEEP_STUCK_AFTER (15m) and converges each — start if missing, leave running ones alone, restart failed ones bounded by TEARDOWN_SWEEP_MAX_RETRIES (3) with TEARDOWN_SWEEP_RETRY_DELAY (5m) between attempts. The retry budget is derived from Temporal visibility — stateless, inspectable in the UI. Every action is idempotent, so overlapping sweeps are harmless.

Durable by history

Workflow history is the source of truth for in-flight state. A worker crash resumes mid-flight — no lease store, no recovery scanner.

Retries are free

ExecuteTask retries 5× with backoff; CompensateTask up to 10×. The workflow code stays a plain state machine.

Waits without polling

Async and app-less steps park on the task_callback signal. An external human or system resumes them exactly when the work is done.

Teardown survives crashes

Cancel is accept-CAS + TeardownWorkflow. The sweep re-converges anything stranded in the accept-vs-start window — bounded retries, then a human.

POSTCancelFulfilment /commerce/fulfilment/requests/{id}/cancel

Stop forward execution — eventually consistent via TeardownWorkflow. compensate=true (default) rolls back completed tasks: COMPENSATING → COMPENSATED (FAILED if a reversal fails). false keeps completed work: CANCELLING → CANCELLED. Terminal → ALREADY_TERMINAL; other mode while tearing down → TEARDOWN_IN_PROGRESS. Tenant-scoped (foreign id reads not-found).
